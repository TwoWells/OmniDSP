# OmniDSP - High-Performance Cross-Platform DSP Library

## Overview

OmniDSP is a high-performance C++ library for Digital Signal Processing (DSP) with Python bindings. Its primary goal is to offer a consistent, easy-to-use API while leveraging highly optimized, platform-specific backends (IntelВ® oneMKL or Apple Accelerate) for maximum performance. It currently includes Fast Fourier Transform (FFT) and Constant-Q Transform (CQT) functionality.

The build system automatically selects the best available backend. If neither is detected, a stub implementation is compiled that throws runtime errors, indicating a functional backend was not available during the build.

## Features

*   **Cross-Platform Performance:** Optimized DSP via native backends.
*   **Platform Abstraction:** Consistent C++ and Python API.
*   **Optimized Backends:** IntelВ® oneMKL or Apple Accelerate framework.
*   **Automatic Backend Selection:** CMake build system handles detection.
*   **Python Bindings:** Seamless integration with NumPy via pybind11.
*   **Precision Support:** `float` and `double` precision.
*   **Transform Types:** 1D Complex-to-Complex (C2C) FFT, Real-to-Complex (R2C) FFT, Complex-to-Real (C2R) IRFFT, Constant-Q Transform (CQT).
*   **Execution Modes:** In-place and out-of-place for C2C FFTs.
*   **Configurable Normalization:** `NormMode::BACKWARD`, `ORTHO`, `FORWARD` for FFTs.
*   **Plan-Based Interface:** `FFTPlan` and `CQTPlan` for efficient repeated transforms.
*   **Convenience Functions:** Simple wrappers for common operations.
*   **Window Functions:** Hann, Hamming, Kaiser, Flat-top.
*   **Modern Build System:** CMake (C++) and `pyproject.toml` with `scikit-build-core` (Python).
*   **Installation Support:** CMake package config files for C++ integration.
*   **Dependency Bundling:** Support for bundling runtime dependencies (e.g., MKL DLLs) into Python wheels using standard tools.
*   **Unit Testing:** GoogleTest (C++) and pytest (Python).

## Prerequisites

1.  **CMake:** Version **3.21** or later.
2.  **C++ Compiler:** A compiler supporting C++17 (e.g., GCC, Clang, MSVC).
3.  **FFT Backend (at least one for full FFT functionality):**
    *   **IntelВ® oneMKL:** Install [IntelВ® oneAPI Base Toolkit](https://www.intel.com/content/www/us/en/developer/tools/oneapi/base-toolkit-download.html). Required if building/running on Windows/Linux, or macOS with MKL preference. Ensure the environment is configured (e.g., via `setvars.bat`/`setvars.sh`).
    *   **Apple Accelerate:** On macOS, included with Xcode and Command Line Tools. Used by default on macOS unless MKL is preferred.
4.  **Python:** Version 3.8 or later (for `os.add_dll_directory` if needed, and general compatibility).
5.  **Python Packages:** `numpy` is required at runtime. Build requirements (`cmake`, `scikit-build-core`, `pybind11`) are specified in `pyproject.toml`. Development requirements are in `requirements-dev.txt`.
6.  **Git:** Required by CMake's `FetchContent` to download GoogleTest and pybind11 during C++ build configuration.

## Building and Installation

There are two primary ways to build the project:

1.  **Building the Python Package (Recommended for Python Users):** This builds the C++ core and the Python bindings, packaging everything into an installable Python wheel. Runtime dependencies (like MKL DLLs) can be bundled into the wheel.
2.  **Building from Source (C++ Development / Usage):** This builds the C++ library, examples, and tests using standard CMake commands, suitable for C++ development or integrating the library into other C++ projects.

### Option 1: Building and Installing the Python Package

The recommended way to install the Python package (`omnidsp_py`) is using `pip`. This process uses `scikit-build-core` and CMake under the hood.

1.  **Prerequisites:** Ensure CMake, C++ Compiler, Python, and Git are installed. If targeting the MKL backend, install the oneAPI Base Toolkit.
2.  **(Required for MKL Backend): Activate oneAPI Environment:** Open a terminal/command prompt and activate the oneAPI environment using the appropriate script (e.g., `setvars.bat` on Windows, `source setvars.sh` on Linux/macOS) provided by your oneAPI installation. **This is crucial** so CMake can find MKL during the build.
3.  **Navigate:** `cd` to the `OmniDSP` project root directory (containing `pyproject.toml`).
4.  **Install via Pip:**
    
    ```bash
    # Install the package into your current Python environment
    pip install .
    
    # For development (editable install):
    # pip install -e .
    
    # For troubleshooting build issues:
    # pip install . -v
    ```
    
    *   This command invokes `scikit-build-core`, which runs CMake (passing `-DOMNIDSP_BUILD_PYTHON_BINDINGS=ON`).
    *   CMake configures the build, selecting the appropriate backend (MKL/Accelerate).
    *   The C++ code and Python bindings are compiled.
    *   The necessary files are packaged into a wheel. CMake's install rules (using `<TARGET_RUNTIME_DLLS:tgt>`) attempt to bundle runtime dependencies like MKL DLLs directly.
    *   The wheel is installed into your `site-packages`.
5.  **Dependency Bundling & Wheel Repair (Optional but Recommended for Distribution):**
    *   While CMake attempts bundling, the most robust way to create distributable wheels with external dependencies included (especially on Windows and Linux) is using dedicated tools like `delvewheel` (Windows) or `auditwheel` (Linux).
    *   **Automated Workflow (using `cibuildwheel`):** Configure `cibuildwheel` in `pyproject.toml` to run `delvewheel repair` or `auditwheel repair` after the initial build. This is ideal for CI/CD pipelines. See the `[tool.cibuildwheel]` section in `pyproject.toml` and the `cibuildwheel` documentation. Remember to configure the `repair-wheel-command` with the correct path(s) for `delvewheel`/`auditwheel` to \*find\* the external libraries (e.g., via `--add-path` for `delvewheel`).
    *   **Manual Workflow:**
        
        ```bash
        # 1. Build the raw wheel
        pip wheel . --no-deps -w dist
        
        # 2. Repair the wheel (Windows example)
        # Ensure MKL DLLs are findable via PATH or --add-path
        # pip install delvewheel
        # delvewheel repair --add-path "C:/path/to/mkl/bin" -w wheelhouse dist/omnidsp_py-*.whl
        
        # 3. Install the repaired wheel from ./wheelhouse
        # pip install wheelhouse/omnidsp_py-*.whl
        ```
        

### Option 2: Building from Source (C++ Focus)

Use this method for C++ development, running C++ examples/tests, or installing the C++ library for use in other CMake projects.

1.  **Prerequisites:** Ensure CMake, C++ Compiler, Git, and optionally the oneAPI Base Toolkit (for MKL) are installed.
2.  **(Required for MKL Backend): Activate oneAPI Environment:** As described in the Python build section, run the `setvars` script if you need the MKL backend.
3.  **Configure using CMake:**
    
    ```bash
    # Create a build directory
    mkdir build
    cd build
    
    # Configure (OMNIDSP_BUILD_PYTHON_BINDINGS defaults to OFF)
    cmake ..
    
    # Optional flags:
    # cmake .. -DCMAKE_INSTALL_PREFIX=/path/to/install -DBUILD_SHARED_LIBS=OFF -DFFT_PREFER_ONEMKL=ON
    ```
    
4.  **Compile:**
    
    ```bash
    # Single-config generator (Makefiles, Ninja)
    cmake --build . --parallel
    
    # Multi-config generator (Visual Studio)
    cmake --build . --config Release --parallel
    ```
    
5.  **Run Tests (Optional):**
    
    ```bash
    # From the build directory
    ctest -C Release --output-on-failure
    ```
    
6.  **Install C++ Library (Optional):**
    
    ```bash
    # Installs headers, libraries, and CMake config files to CMAKE_INSTALL_PREFIX
    cmake --install . --config Release
    ```
    

### Option 3: Using `build.bat` (Windows Convenience)

A `build.bat` script is provided for convenience on Windows. It wraps the CMake configure, build, test, and optional Python wheel build/repair steps.

1.  **Prerequisites:** As above. Ensure Python and necessary dev packages (`requirements-dev.txt`) are installed if using the Python options.
2.  **(Required for MKL Backend):** The script attempts to find and run `setvars.bat` automatically based on `ONEAPI_ROOT` or a default path. Ensure oneAPI is installed correctly.
3.  **Run the script:** Open a Command Prompt or PowerShell.
    
    ```bash
    # Build C++ only (Release) and run tests
    build.bat
    
    # Build C++ (Debug), prefer MKL, skip tests
    build.bat -t Debug -b mkl --no-tests
    
    # Build C++ and Python wheel (Release), run tests
    build.bat --python ON
    
    # See help for all options
    build.bat --help
    ```
    
4.  If `--python ON` is used, the script attempts to build the wheel using `pip wheel` and repair it using `delvewheel` (relying on the `PATH` set by `setvars.bat`). Repaired wheels appear in `wheelhouse/`.

## Using the Library

### From C++

After installing OmniDSP using `cmake --install`, other CMake projects can use `find_package`:

```cmake
# In your project's CMakeLists.txt
find_package(OmniDSP 1.0.0 REQUIRED)
# ...
target_link_libraries(my_target PRIVATE OmniDSP::omnidsp)
```

```cpp
// In your C++ code
#include <OmniDSP/omnidsp.h>
#include <vector>
#include <complex>

int main() {
    std::vector<float> signal = {1.f, 0.f, 0.f, 0.f};
    std::vector<std::complex<float>> spectrum;
    OmniDSP::rfft(signal, spectrum);
    // ... use spectrum ...
    return 0;
}
```

### From Python

After installing the package via `pip install .` (or installing a repaired wheel):

1.  **IMPORTANT: Ensure Runtime Dependencies are Found (Especially MKL):**  
    The installed Python module (`omnidsp_py`) needs access to its backend runtime libraries (e.g., MKL DLLs on Windows/Linux) at runtime. Choose **one** method:
    *   **Bundled Dependencies (Recommended):** If you installed a wheel repaired by `delvewheel` or `auditwheel`, the necessary libraries should be bundled \*inside\* the wheel and loaded automatically. This is the ideal scenario.
    *   **Environment Setup (Standard):** Activate the Intel oneAPI environment (`setvars.bat` / `setvars.sh`) in your terminal **before** launching Python, Jupyter, or your IDE. This adds the required DLL directories to the system `PATH` (or equivalent).
    *   **`os.add_dll_directory` (Windows Python 3.8+ Fallback):** If the above methods fail (e.g., `ImportError: DLL load failed`), you can \*programmatically\* add the paths to the MKL (and potentially Compiler/TBB) `bin` directories at the \*start\* of your Python script before importing `omnidsp_py`. See example code in the original README or online resources. Adjust paths to match your oneAPI installation.
    *   **`.env` File (IDE-Friendly):** Create a `.env` file in your workspace root and add the MKL/Compiler/TBB bin directories to the `PATH` variable (see original README example). VS Code's Python extension often loads this.
2.  **Import and Use:**
    
    ```python
    import numpy as np
    import omnidsp_py # Module name defined in CMake/bindings
    
    # Example using rfft (double precision)
    N = 16
    signal = np.cos(2.0 * np.pi * 3.0 * np.arange(N) / N).astype(np.float64)
    
    try:
        spectrum = omnidsp_py.rfft_double(signal)
        print(f"Spectrum shape: {spectrum.shape}") # N/2 + 1
        print(spectrum[:5])
    
        # Reconstruct
        reconstructed = omnidsp_py.irfft_double(spectrum)
        print(f"Reconstructed close to original: {np.allclose(signal, reconstructed)}")
    
    except ImportError as e:
         print(f"ImportError: {e}")
         print("Ensure omnidsp_py is installed and runtime dependencies (like MKL DLLs) are accessible.")
         print("Try activating oneAPI environment or using os.add_dll_directory (Win Py3.8+).")
    except Exception as e:
        print(f"An error occurred: {e}")
    ```

## Notes / Limitations

*   **FFT Normalization:** Convenience functions use `NormMode::BACKWARD`. Use `FFTPlan` via C++ or Python for `ORTHO` or `FORWARD` modes.
*   **IRFFT Input:** Input to `irfft` requires Hermitian symmetry for real output.
*   **Accelerate Real FFT Length:** Requires power-of-2 length `N` for `Domain::REAL` transforms. oneMKL is more flexible.
*   **Thread Safety:** Different `FFTPlan` objects are thread-safe. Using the \*same\* plan concurrently on \*different\* data should be safe (check backend docs). Modifying the same plan concurrently is unsafe.
*   **Supported Transforms:** 1D C2C/R2C/C2R FFT, CQT. Multi-dimensional FFTs not implemented.
*   **Runtime Dependencies:** End-users need the runtime libraries for the backend OmniDSP was built with (MKL or Accelerate). Ensure proper environment setup or use wheels with bundled dependencies.
